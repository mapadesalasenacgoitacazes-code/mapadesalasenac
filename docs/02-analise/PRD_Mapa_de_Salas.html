1. Visão Geral do Produto

O que é: Um sistema web para gerenciar a alocação de salas de cursos do SENAC Minas, substituindo a planilha Excel atualmente utilizada pela supervisão.

Por que existe: A planilha atual apresenta problemas reais: erros de fórmula (#N/D), dificuldade de filtrar informações cruzadas (ex: "por quais salas a turma X já passou?"), risco de conflitos não detectados (duas turmas na mesma sala/turno) e limitação de visualização por ano.

Para quem: Equipe administrativa do SENAC (Supervisão, Suporte, Docentes). Não é voltado para alunos.

Proposta de valor em uma frase: Trocar a planilha por um sistema que impede conflitos de sala automaticamente, calcula progresso dos cursos em tempo real e permite consultar o histórico de qualquer turma em segundos.

2. Problema a Resolver

Problema Impacto hoje Como o sistema resolve
Conflitos de sala Já aconteceu de alocar 2 turmas na mesma sala/turno. A supervisão detecta visualmente, mas depende de atenção humana. UNIQUE constraint no banco: impossível fisicamente salvar reserva duplicada. Sistema avisa e sugere salas disponíveis.
Filtro "salas por turma" inexistente Para saber por quais salas a Gastronomia passou, precisa buscar manualmente na planilha inteira. View vw_historico_salas_turma retorna a resposta em 1 clique por código de turma.
Erros de fórmula Linhas vazias geram #N/D. Poluem a planilha e confundem a leitura. Dados em banco relacional, sem fórmulas. Campos obrigatórios garantidos por constraint.
Limitação de ano A planilha é "ANO 2026". Cursos que cruzam dez/jan ficam partidos entre duas planilhas. Sistema trabalha com datas reais (início/término), sem limite de ano.
Cálculo de término manual Quando um dia é cancelado, alguém precisa lembrar de ajustar a data de término. RN01 automatizada: dia cancelado recalcula término. Progresso atualizado em tempo real.

5. Regras de Negócio Críticas

Código Regra Descrição Implementação técnica
RN01 Dia sem aula estende término Dia cancelado (falta, evento, etc.) adiciona +1 dia útil ao término do curso, pulando feriados e dias não letivos. Tabela dia_cancelado + trigger ou lógica na aplicação que recalcula data_termino_calculada.
RN02 Conflito impossível Duas turmas não podem ocupar a mesma sala + turno + dia. UNIQUE(id_sala, data, turno) na tabela reserva. Banco rejeita antes do código.
RN03 Turma multi-sala Uma turma pode usar salas diferentes no mesmo dia (turnos diferentes). Ex: Gastronomia manhã na cozinha, tarde no lab. Reserva é por (turma, sala, data, turno) - não por dia inteiro. Permite N reservas por turma por dia.
RN04 Progresso automático % = (dias realizados / dias previstos) x 100. Dias cancelados não contam como realizados. View vw_progresso_turma calcula sob demanda com base nas reservas e cancelamentos.
RN05 Manutenção bloqueia sala Sala em manutenção não aceita novas reservas. Reservas futuras existentes geram alerta. Check no momento da reserva: IF sala.status = 'Manutenção' THEN rejeitar.
RN06 Sem limite de ano Cursos podem cruzar virada de ano sem quebrar. Sem particionamento por ano. Datas são campos DATE puros.

8. Riscos e Mitigações

Resistência à mudança
A equipe está acostumada com a planilha Excel há anos. Podem rejeitar o sistema novo.
Mitigação: manter a interface parecida com a planilha (formato calendário/Gantt). Treinar com a planilha real como referência.
Cálculo de término incorreto
A regra RN01 (+1 dia por cancelamento) é complexa: precisa pular feriados, fins de semana e dias não letivos da turma.
Mitigação: criar testes automatizados exaustivos para esse cálculo. Validar com dados reais de 2026.
Escopo cresce durante o projeto
A cliente pode pedir funcionalidades novas no meio do desenvolvimento (integração com chamados, app mobile, etc.).
Mitigação: PRD aprovado com MoSCoW claro. Novos pedidos entram como Won't Have até a V2.
Dados da planilha atual
Se a cliente quiser migrar os dados históricos da planilha, o formato pode ser inconsistente.
Mitigação: lançar o sistema limpo (dados novos). Histórico fica na planilha como consulta. Migrar apenas se houver tempo.

11. Stack Tecnológica (Sugestão)

Stack definida pelo professor. PHP + MySQL é a combinação padrão do curso e amplamente usada no mercado para sistemas web internos.

Camada Tecnologia Justificativa
FrontendHTML + CSS + JavaScript + Bootstrap 5Interface responsiva, componentes prontos (tabelas, modais, formulários)
BackendPHP 8+Definido pelo professor. Linguagem do curso, boa documentação em PT-BR
Banco de dadosMySQL 8Compatível com o schema.sql já criado. PDO para conexão segura
AutenticaçãoPHP Sessions + password_hash()Nativo do PHP, sem dependência externa. Senhas com bcrypt
